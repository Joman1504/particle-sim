GPU-Accelerated 2D Particle Simulation

Real-time CUDA physics + OpenGL rendering with CPU/GPU comparison

CS4220 — GPU Computing
Author: _______________________
Hardware: Intel i7-13700K · RTX 4080 (sm_89) · CUDA / GLFW / GLAD

Project idea

Build an interactive many-particle toy simulation in normalized device coordinates (NDC): gravity, bottom respawn, wall bounces, mouse attraction, and collisions with a movable triangle obstacle.

The core question: Can massively parallel GPU execution overcome the scalability bottleneck of sequential CPU particle updates? The app toggles between a sequential CPU path and a CUDA one-thread-per-particle GPU path (press G) to compare FPS, frame time, and maximum real-time particle count directly.

Interactive Features

  • Physics — gravity, elastic wall bouncing (restitution 0.8), bottom-exit respawn with randomized re-entry.
  • Obstacles & interaction — draggable triangle with circle–edge collision; right-mouse inverse-square attraction force.
  • Runtime controls — zero-gravity (Z), slow-motion (M), particle count presets 10k–10M (1–4).

Methods / pipeline

  • CPU: updateParticlesCPU — nested loops, same integrator & bounds as GPU.
  • GPU: CUDA kernels, block size 256; one thread updates one particle per frame.
  • Interop: CUDA registers the OpenGL position VBO (cudaGraphicsGLRegisterBuffer); each frame positions are written on device, avoiding host staging.
  • Collision: circle vs triangle — closest point on edges, outward normal, position correction + velocity reflection (shared host/device logic).

Implementation snapshot

updateKernel <<< grid, 256 >>>(d_parts, n, dt, gravityY, spawnSpeed, seed);

Key takeaway

Massively parallel CUDA maps naturally to independent particles; pairing it with CUDA–OpenGL buffer interop keeps the render path cheap. The GPU path is the practical way to sustain high N at interactive frame rates.
